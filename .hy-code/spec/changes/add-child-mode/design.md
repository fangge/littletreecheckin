## Context

应用由家长和孩子共用同一账号，当前没有任何权限分层机制。孩子在使用时可能误触编辑目标、访问家长审核等敏感功能。需要一个轻量级的"儿童模式"来限制孩子的操作范围，同时保证家长可以随时通过密码验证切换回完整模式。

## Goals / Non-Goals

- **Goals**:
  - 儿童模式下隐藏任务编辑、目标添加、家长中心等敏感功能
  - 模式切换（开启/关闭）均需账户登录密码二次确认
  - 儿童模式状态在页面刷新后持久化
  - 提供明显的视觉提示，让孩子知道当前处于儿童模式
- **Non-Goals**:
  - 不引入多用户角色系统（家长/孩子分账号）
  - 不修改数据库结构
  - 不在后端强制执行儿童模式权限（仅前端 UI 控制 + 路由守卫）

## Decisions

### 决策 1：状态存储方案 — localStorage 持久化

**选择**: 将 `isChildMode` 存储在 `localStorage`（key: `child_mode`），在 `AuthContext` 中读取和管理。

**理由**:
- 无需修改数据库结构，实现成本低
- 状态与设备绑定，符合"家长在自己设备上开启儿童模式给孩子使用"的场景
- 登出时清除该状态，避免状态残留

**替代方案**: 存储在数据库用户表中 → 实现复杂，且跨设备同步对本需求意义不大，暂不采用。

### 决策 2：密码验证方案 — 新增专用验证接口

**选择**: 在后端新增 `POST /api/v1/auth/verify-password` 接口，接收密码并返回验证结果，不生成新 token。

**理由**:
- 复用登录接口会产生副作用（如更新 token、触发登录日志等）
- 专用接口语义清晰，职责单一
- 接口受 JWT 中间件保护，确保只有已登录用户可调用

**替代方案**: 复用 `/api/v1/auth/login` 接口 → 有副作用风险，不推荐。

### 决策 3：UI 控制方案 — 条件渲染

**选择**: 在各组件中通过 `const { isChildMode } = useAuth()` 获取状态，使用条件渲染（`{!isChildMode && <Component />}`）隐藏受限 UI 元素。

**理由**:
- 实现简单直观，符合 React 惯用模式
- 无需引入额外的权限库或 HOC

### 决策 4：路由守卫 — 重定向至 Dashboard

**选择**: 在 `App.tsx` 中为受限路由添加守卫组件，儿童模式下访问时重定向至 `/dashboard`。

**理由**:
- 防止孩子通过直接输入 URL 绕过 UI 限制
- 重定向至 Dashboard 是最自然的回退页面

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| 前端状态可被技术用户绕过（如修改 localStorage） | 当前场景为家庭内部使用，孩子通常不具备此能力；如需更强安全性，可后续在后端添加模式验证 |
| 多标签页状态不同步 | 通过 `storage` 事件监听 `localStorage` 变化，同步更新 `isChildMode` 状态 |
| 登出后儿童模式状态残留 | 在 `logout` 方法中清除 `localStorage` 中的 `child_mode` 键 |

## Migration Plan

- 现有用户升级后默认 `isChildMode = false`（`localStorage` 中无该键时默认为 `false`）
- 无需数据迁移
